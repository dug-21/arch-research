## Part A — New hypotheses

### H25 (S1) — Real-time guardrail/steering layer over the live observation buffer

- **Statement:** an in-server Rust automata stack (`RegexSet`/`aho-corasick` as the fast lexical tier; tree-sitter incremental parse as an optional structural tier) could enhance `context_cycle` of Unimatrix by streaming every observation-buffer append through compiled pattern banks and mapping each match class to a **graduated action ladder** — *nudge* (advisory surfaced into the agent's next context) → *course-correct* (structured steering message) → *throttle* (hold/rate-limit the run) → *kill* (terminate the agent) — so course-correction happens **while the run is happening**, not post-hoc at review.
- **Mechanism (falsifiable core):** the `regex` crate is RE2-lineage — finite-automata, guaranteed linear time, no backtracking — and `RegexSet` compiles thousands of patterns into one automaton evaluated in a single pass; `regex-automata`'s lower-level DFAs support **streaming with match state persisting across appends**, so cost per append is proportional to the appended bytes, not the buffer size. Because these are pure-Rust crates inside the server binary, the check sits inline on the observation write path the server already owns — no external process, no network hop, deterministic ms-scale worst case. Severity is per-pattern metadata: match → severity class → ladder rung is a table lookup, not a judgment call. This unifies pass-1 H2/H3 (structural take) and H26 (safety take) as consumers of one layer.
- **Target:** `context_cycle` observation path — a new in-server middleware capability; downstream, the factory's ability to run autonomously (an in-run governor is what unattended cadence is gated on).
- **Class:** non-obvious (pass 1 couldn't see it only because the buffer-access wall hid it; the *graduated actuator* — deterministic detection driving proportionate intervention — is the genuinely new part).
- **Level-up vs linear:** **level-up.** Review can only report; a live layer can intervene. It converts the platform from a recorder into an actuator — a different capability class, and the specific step-function the owner named (real-time feedback during a run).
- **Cheapest test:** replay 2–3 historical cycle buffers append-by-append through a standalone Rust prototype (~50 patterns derived from lessons-learned nodes + known runaway/thrash signatures); measure per-append latency and log where each ladder rung *would* have fired; compare against what the human actually did mid-run. One "would have course-corrected materially earlier than the human noticed" hit, at sub-ms append cost, confirms the mechanism.
- **Key assumption:** run pathologies worth intervening on have **lexically detectable precursors early enough** for correction to matter. **Biggest risk:** the actuator half, not the matcher — detection is trivial, but *delivering* a nudge into a running agent's context requires a feedback channel into the agent loop (owner-liftable, but it is genuinely new platform surface), and a mis-timed injected correction can derail a healthy run. False-positive cost scales with rung severity — the graduated ladder is load-bearing engineering, not decoration.

### H26 (S2) — Abuse-pattern watchdog with graduated agent termination

- **Statement:** an in-server `aho-corasick`/`RegexSet` watchdog could enhance run safety of Unimatrix by matching the live transcript against **two deliberately separated signature banks** — **security** (a manipulated/compromised agent: prompt-injection markers arriving in untrusted content, "ignore previous instructions" families, exfil-shaped command sequences, encoded-payload signatures) and **reliability/cost** (a stuck/runaway agent: repeated identical tool-call+error n-grams, retry storms, output-length blowup) — and driving the top rungs of H25's ladder (throttle → kill).
- **Mechanism (falsifiable core):** `aho-corasick` matches enormous *literal* banks in one streaming pass at ~bytes/cycle cost (published injection corpora are literal or near-literal — exactly its workload); `regex-automata` overlapping search catches interleaved signatures. The reliability bank exploits a specific structural property: **stuck loops are literally repetitive** — the same byte sequences recur — so deterministic detection works there without any semantics. The two banks are separated because their false-positive economics differ: wrongly killing a compromised agent costs nothing; wrongly killing a healthy 3-hour run burns real budget — so reliability enters the ladder at *throttle*, security may escalate to *kill* at high-confidence signatures. Graduated response, never a binary axe.
- **Target:** run lifecycle — an agent-termination/throttle hook (owner-liftable endpoint); directly feeds the factory's **budget-metering** prerequisite (memory: autonomous scheduled scanning is gated on exactly this class of governor).
- **Class:** adjacent — this is the IDS/DPI design center of the automata wing (scout candidate #3) transplanted from network packets to agent transcripts.
- **Level-up vs linear:** **level-up for autonomous operation** (unattended runs require something watching that can pull the plug; today that something is the human); linear for attended runs.
- **Cheapest test:** reliability bank first, because history is self-labeling: scan past cycle buffers for repeated tool-call/error n-grams; verify known-bad runs (any the human killed or that visibly thrashed) are flagged and healthy runs are not. Security bank: replay a public injection corpus (e.g. jailbreak string collections) through a mock transcript; measure hit rate and clean-transcript false positives.
- **Key assumption:** the deterministically catchable fraction of abuse is worth catching — obvious abuse is lexical; subtle manipulation is semantic and this layer will never see it (the owner's stated limit, kept explicit). **Biggest risk:** two-headed — (a) *false-negative complacency*: a watchdog that catches the easy 60% gets mistaken for coverage, and an actually-adversarial injector trivially paraphrases past a literal bank (the security bank is a tripwire, not a wall); (b) *false-positive kill* on long healthy runs — mitigated by the ladder and by keeping the two banks' thresholds independent.

### H27 (S3) — Learn-then-enforce edge grammar (graph quality/consistency)

- **Statement:** Unimatrix's graph consistency could be enhanced by **inferring the graph's implicit edge schema by enumeration** — tabulate every `(source-category, edge-type, target-category)` triple that actually occurs across the ~100-node graph — reconciling that table against the sanctioned D8 rule set, **surfacing existing violations in the same pass**, then compiling the sanctioned triple set into an in-server write-time validator on `context_edge`: H17's tag grammar, but for topology.
- **Mechanism (falsifiable core):** at current scale (6 edge types × ~6 categories) the entire schema space is a few hundred cells — the "learn" step is a **single graph export + group-by, exhaustive enumeration, zero ML**. The double-duty property is the core: the *same* enumeration that infers the schema *is* the violation report — any occurring triple outside D8's table is either a real violation or an undocumented convention, and both are findings. Enforcement afterward is a constant-time set-membership check per edge write (it doesn't even need a regex engine — the family's *thinking*, not its heaviest tool). **Reuse-first:** the prior art for graph-shape constraints is **SHACL/ShEx** from the RDF world — mine their constraint taxonomy (node-kind constraints, cardinality, closed shapes: "factory-plane edges only factory→factory" is a closed-shape rule; "Supersedes is system-only" is a provenance constraint) for what a v2 validator should express beyond bare triples; steal the vocabulary, skip the RDF stack.
- **Where it runs:** **both** — the periodic sweep first (it is the cheapest test and catches drift introduced by corrections/migrations), then write-time validation (catches new drift at zero marginal cost). The sweep also re-validates the validator whenever D8 evolves.
- **Target:** `context_edge` write path + graph-consistency maintenance; moves the D8 edge rules from prompt-discipline to mechanical enforcement.
- **Class:** adjacent (the enumeration-not-ML sharpening plus the SHACL transfer is the non-trivial move).
- **Level-up vs linear:** linear (quality/discipline) — with a small step-function at multi-goal scale: cross-goal compounding is only trustworthy if the graph's topology is mechanically guaranteed consistent, and that guarantee gets more valuable per node added.
- **Cheapest test:** one afternoon: export the graph (`context_graph` subgraph / DB read), group-by the triples, diff against the D8 table. Any violation found is immediate retroactive value; a clean result still yields the compiled whitelist for the write-time guard.
- **Key assumption:** edge conventions are stable enough that a whitelist doesn't fight legitimate schema evolution. **Biggest risk:** enforcement rigidity — D8 is versioned method, not eternal truth; a hard reject on a triple the method just legitimately added creates a blocked-curator failure mode. The validator must be config-as-data (the D8 table as a file the owner edits), warn-before-reject, with an explicit override path.

### H28 — One pattern-bank substrate, compiled from the graph itself (whitespace)

- **Statement:** a single in-server **pattern-bank service** — compile-once `RegexSet`/`aho-corasick` banks, hot-reloadable, populated **from the graph itself** (lesson-learned/finding nodes carrying a structured `signature:` field) — could enhance the entire `context_*` surface by making every pattern consumer (H25 guardrails, H26 watchdog, H11 review bank, H13 PII coverage, H15 reverse-search, H17 tags) sockets of one substrate, and by making stored knowledge **executable**: filing a lesson-learned mechanically upgrades the live defenses on the next bank compile.
- **Mechanism (falsifiable core):** `RegexSet`'s defining economics — n patterns, one automaton, one pass, marginal cost of pattern n+1 ≈ 0 — mean consumers share a scan instead of each paying for one; and Rust regex compilation of thousands of patterns is sub-second, so **compile-on-write is inline-feasible**: `context_store` of a signature-bearing node triggers bank recompilation, making knowledge→enforcement latency exactly one write. This closes the factory's reflexive loop *lexically*: the compounding-asset objective stops being metaphorical — the KB literally gets sharper as a watcher with every node.
- **Target:** platform architecture (a server-internal capability all pattern hypotheses ride on); the compounding-asset objective as a mechanism rather than a metric.
- **Class:** non-obvious (whitespace/composition — the architecture move underneath the features, visible only once the Rust-in-binary correction lands).
- **Level-up vs linear:** **level-up** — changes the KB from passive (queried when an agent thinks to ask) to active (watching everything that streams past), and it compounds: every stored signature raises coverage at zero marginal scan cost.
- **Cheapest test:** a ~200-line standalone Rust binary: read signature-tagged nodes from a graph export, compile a `RegexSet`, scan a historical buffer; measure compile time at 100 / 1k / 10k patterns and scan throughput. Confirms the economics before any server surgery.
- **Key assumption:** a useful fraction of lessons-learned admit *distinctive* lexical signatures (inherited from H15 — still the hard part). **Biggest risk:** substrate-before-consumers inversion — building the platform layer before any single consumer is proven is classic premature infrastructure. Triage framing: fund H28 as the **refactor when the second pattern consumer ships**, not as the first consumer's foundation.

### H29 — The firewall as an automaton: grade-transition state machine

- **Statement:** a tiny in-server finite-state machine could enhance `context_tag`/`context_correct` of Unimatrix by encoding the firewall's **legal grade transitions** as an automaton — states `{missing, claimed, partial, proven}`; the `proven`-entering edge accepts **only** when the write is a `context_correct` attaching a `proven_by` artifact ref, never a bare `context_tag` (D7 verbatim) — making the factory's most load-bearing rule mechanically unviolable instead of prompt-enforced.
- **Mechanism (falsifiable core):** the family's primitive (a DFA) applied to *lifecycle* rather than text: transition legality is a constant-time table lookup keyed on `(current-grade, requested-grade, write-verb, has-proven_by)`. Complements H17 exactly: H17 validates tag **shape**; H29 validates tag **history**. In-server Rust makes it a guard clause on two existing endpoints — near-zero integration surface.
- **Target:** `context_tag` / `context_correct`; the D7 firewall itself. Notably valuable *now* because attribution is not yet trustworthy (D6: writes record `created_by: anonymous`) — mechanical guards matter most precisely while audit is weakest.
- **Class:** adjacent (automata-thinking transferred from text to state lifecycle).
- **Level-up vs linear:** linear — but with outsized leverage per line of code, because the rule it hardens underwrites every grade in the graph.
- **Cheapest test:** audit history first: walk existing grade-bearing nodes' chains/tags for any past transition the table would have rejected (a nonzero count proves the need and pays for the test); then a ~50-line guard + a unit-test suite over the transition table.
- **Key assumption:** illegal transitions occur, or will at multi-agent/multi-goal scale. **Biggest risk:** near-zero technical; the real risk is over-rigidity on *legitimate* moves — e.g. demoting `proven`→`partial` when an artifact rots must stay legal, and the transition table must be authored from D7 and owner-reviewed, not guessed by the implementer.

## Part B — Re-scored / revived priors (what the corrected surface changes)

**H2 — revived (wall lifted + effort drop).** Pass-1's biggest risk was "transcript format is owned by the platform and may change without notice — a maintenance liability coupled to an unstable upstream." The owner *is* the platform: the grammar and the buffer format can co-evolve, and stronger — the format can be made **grammar-first** (designed for parseability, or emitted through a constrained emitter), deleting the risk rather than managing it. tree-sitter's first-class Rust bindings put the parser inside the server binary. H2 is now the *structural tier* of H25's stack rather than a standalone bet.

**H3 — revived (its kill condition was exactly the liftable wall).** Pass 1 said it plainly: "if the buffer is only readable at review time… the hypothesis dies on platform access, not on the technology." Corrected read: the server is where appends land, so the incremental parse sits precisely at the point of accumulation — no mid-cycle *access* is even needed; the intelligence is already co-located with the stream. Restated: tree-sitter's in-server incremental re-parse maintains a live typed event tree as the buffer grows, and H3 becomes H25's structural-escalation tier (lexical bank catches known signatures; the live tree catches *shape* pathologies — phase stalls, gate-span anomalies — the bank can't express).

**H16 — revived (effort class collapses).** Pass-1 parked it as "a platform feature request on Unimatrix, not a bolt-on — delivery depends on the Unimatrix roadmap." The owner owns the roadmap, and the mechanism needs **no engine dependency at all**: product-BFS of a tiny DFA (6-label alphabet) with the existing graph store, ~100 lines of Rust as a new `context_graph` mode. Effort re-reads from roadmap-blocked to small in-server feature; the fit/value case from pass 1 stands as written.

**H1 / H9 (and the H1/H9/H20 bake-off) — effort drop + bake-off narrows.** The content-grammar slot was discounted partly on integration (external tooling around the MCP boundary). Corrected: **ast-grep IS Rust** (`ast-grep-core` is an embeddable crate) and tree-sitter binds natively — the write-time validator/extractor lives inside `context_store`/`context_correct`, no external process. The pass-1 substrate collision (fund one of H1/H9/H20) **narrows to tree-sitter vs ast-grep-core**, both in-binary: pika has no production Rust path (its ecosystem-thinness risk now includes "no in-server route"), and while `pest` gives PEG in Rust, it doesn't carry pika's specific recovery property — H20's distinguishing mechanism loses its deployment story.

**H11 — engine correction + subsumption.** Doesn't need Hyperscan FFI at today's scale: `RegexSet` + `aho-corasick` in-binary handle hundreds-to-thousands of patterns; Hyperscan/vectorscan via FFI only if SIMD-scale throughput ever demands it. Effort drops to in-server, and H11 becomes the *post-hoc mode* of the same bank H25 runs live — one investment, two consumption points (during-run and at-review).

**H13 — REFRAMED (not greenfield).** A PII scanner **already exists in the server**. The hypothesis is no longer "add a write-time safety scan" but: *could a `RegexSet`/`aho-corasick` multi-pattern bank improve the existing scanner's **coverage** (secret/credential families beyond PII — the gitleaks/trufflehog corpus) and **throughput** (one-pass compiled bank)?* Cheapest test changes accordingly: first characterize the existing scanner (engine, pattern set, recall on a labeled corpus), then measure the delta. New risk: if the existing scanner is already `regex`-crate-based, this collapses to a **pattern-set contribution, not a technology adoption** — worth knowing before any proof-goal is scoped.

**H17 — effort drops to trivial; bundle.** `regex` crate in-binary, anchored matches inline on write. Pass-1 already flagged it "almost too small to be a proof-goal by itself"; now it is unambiguously a bundle item — natural bundle: H17 (tag shape) + H29 (grade transitions) + H27's write-time guard (edge triples) = one "write-path deterministic guards" package, three small tables behind one pattern of enforcement.

**H12 — effort drop, core risk unchanged.** The exact-identifier channel runs in-server beside the embedding query (one process, one call). The load-bearing risk — rank fusion worsening the common case — is untouched by the substrate correction.

**H14 — engine substitution re-read.** Pure-Rust `regex`/`aho-corasick` first; vectorscan FFI only at SIMD scale. This *lowers* integration cost and simultaneously *lowers urgency* — the Rust crates likely suffice at shd-006's current rule counts, reinforcing pass-1's premature-optimization caution.

**H15 — matcher side collapses into H28; wall liftable; hard part unchanged.** The engine story is now in-binary `aho-corasick` streaming (an H28 consumer), and the live-session stream is an owner-liftable endpoint rather than a wall. The signature-derivation problem — pass-1's honestly-flagged hard part — is unchanged and remains the actual bet.

**H19 — partial re-read only.** The effort caveat softens on both counts (PEG via `pest` is in-binary; the owner owns the platform), but the load-bearing risk — agents underperform on invented syntax vs the tool-call idiom they're trained on — is untouched. Verdict picture moves on effort, not on the core doubt.

## Flags for the leader / triage (pass-2 delta)

1. **One automata stack, many consumers:** H25/H26/H11/H15/H13-reframed/H28 are all the same in-server `regex`/`RegexSet`/`aho-corasick` substrate pointed at different streams. Triage should price the substrate once (H28 is its explicit architecture form) and treat the rest as consumers — not six independent efforts.
2. **The real new platform surface is the actuator, not the matcher:** every intervention hypothesis (H25/H26, revived H3) is trivial on the detection side and novel on the *feedback-channel-into-a-running-agent* side. That channel is owner-liftable but is the genuine engineering unknown; a proof-goal that only proves matching has dodged the question.
3. **Write-path guard bundle:** H17 + H29 + H27(write-time) are three small tables behind one enforcement pattern — a natural single proof-goal.
4. **False-positive economics are the recurring design constraint:** graduated ladders (H25/H26), warn-before-reject (H27), owner-authored transition tables (H29). Deterministic ≠ safe-to-act-on; the action policy is where these live or die.
5. **H13 needs a characterization step before any scope:** what the existing PII scanner is built on decides whether the hypothesis is a technology adoption or a pattern-set PR.
6. **Firewall:** all of the above are `claimed` conjectures. Pass-2 additions: 5 new (distribution: 3 adjacent, 2 non-obvious/whitespace); revivals move *structure and effort reads only*, no grades moved, nothing proven.
